Phase 1

Dataset Construction

The training set is fully synthetic — no real patient data is ever used. A frontier model (Claude Sonnet or GPT-4o) acts as the "synthetic annotator," generating gold outputs using the exact NeuroBee system prompt from src/app/api/insights/route.ts.

Validation Filter (critical)

def validate_example(output_json):
    required = {"reassurance", "observation", "normalization", "guidance", "visualObservation"}
    if not required.issubset(output_json.keys()): return False
    if not isinstance(output_json["guidance"], list) or len(output_json["guidance"]) != 3: return False
    banned = ["has autism", "diagnos", "definitely has", "confirms autism"]
    text = json.dumps(output_json).lower()
    if any(b in text for b in banned): return False
    for field in ["reassurance", "observation", "normalization"]:
        if len(output_json[field].split()) > 120: return False
    return True

Model Loading

from unsloth import FastLanguageModel

MODEL_ID = "unsloth/gemma-4-4b-it"   # or "unsloth/gemma-2-4b-it" as fallback

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name=MODEL_ID,
    max_seq_length=2048,   # system+user+assistant fits comfortably
    dtype=None,             # auto: bf16 on A100, fp16 on T4
    load_in_4bit=True,     # QLoRA — mandatory for T4
)

target modulesq/k/v/o_proj + gate/up/down_projFull attention + MLP coverage for instruction following

gradient checkpointingunsloth30% VRAM reduction — mandatory for T4
 
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_alpha=32,
    lora_dropout=0.05,
    bias="none",
    use_gradient_checkpointing="unsloth",
    random_state=42,
)

Phase 3

Multimodal Strategy

Gemma 4 E4B is natively multimodal (SigLIP vision encoder + language decoder). The recommended approach for NeuroBee is text-only fine-tuning. This is the correct architectural decision, not a compromise:

Reason 1

Vision encoder is already capable

SigLIP pretraining gives the model the ability to describe gaze, posture, and engagement from video frames without task-specific fine-tuning.

Reason 2

Frames can't be in training data

Real camera frames vary per child and can't appear in a synthetic training set without actual clinical data. The visualObservation field benefits least from fine-tuning.

Reason 3

Task-specific knowledge is in the decoder

IAP clinical language, RBSK references, warm Indian English tone, and the JSON output schema live entirely in the language decoder — that's what needs tuning.

Future Work

Multimodal fine-tuning (Phase 2)

Once text-only tuning is validated, use HuggingFace transformers + PEFT with AutoProcessor for interleaved text+image batches. Requires custom data collation and an A100.

Implementation: Training examples have visualObservation: null. During inference, the production system sends real frames — base vision capability handles this correctly without task-specific training.

Phase 4

GGUF Export for LM Studio

Step 1 — Save merged weights from Colab

# After training completes:
model.save_pretrained_merged(
    "neurobee-merged",
    tokenizer,
    save_method="merged_16bit",   # required input for llama.cpp converter
)
# Download neurobee-merged/ to Google Drive or local machine

Step 2 — Convert to GGUF (on your Windows dev machine)

# One-time setup
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
pip install -r requirements.txt
cmake -B build && cmake --build build --config Release

# Convert HuggingFace weights → GGUF (unquantized f16)
python convert_hf_to_gguf.py \
    /path/to/neurobee-merged \
    --outfile neurobee-gemma4-f16.gguf \
    --outtype f16

# Quantize to Q4_K_M (recommended for 16 GB RAM machines)
./build/bin/llama-quantize neurobee-gemma4-f16.gguf neurobee-gemma4-Q4_K_M.gguf Q4_K_M

Phase 6

Integration into NeuroBee

# Copy your quantized file there:

In LM Studio UI: load the model → set context length to 2048 → enable GPU offloading → Start server on port 1234.

Step 2 — One env var change (zero code change)

# .env.local
LMSTUDIO_URL=http://localhost:1234
LMSTUDIO_MODEL=neurobee-gemma4-Q4_K_M   # was: google/gemma-4-e4b

Step 3 — Simplify the system prompt (optional)

The fine-tuned model has internalized the JSON schema, no-diagnosis rules, and Indian context. The system prompt can be shortened from ~400 tokens to ~60 tokens, reducing latency:

// Minimal system prompt for the fine-tuned model:

3-Week Execution Timeline

Week 1 — Dataset

Build & generate training data

  • Days 1–2: Write dataset_builder.py + synthetic_annotator.py
  • Day 3: Generate 1,050 raw examples via Claude API (~$2–3 cost)
  • Day 4: Run validation filter; inspect 50 random examples manually
  • Day 5: Create train/val/test splits; human review of 50 test examples
Week 2 — Training & Export

Fine-tune and convert

  • Days 1–2: Open Colab; install Unsloth; run neurobee_finetune.ipynb
  • Day 3: Monitor eval_loss; adjust LR if needed; download merged weights
  • Day 4: Run convert_hf_to_gguf.py + llama-quantize on Windows machine
  • Day 5: Load Q4_K_M into LM Studio; run 50 test prompts; score vs baseline
Week 3 — Evaluation & Rollout

Validate and ship

  • Days 1–2: Run human evaluation rubric on test set (5 dimensions × 50 examples)
  • Day 3: If eval passes → update .env.local LMSTUDIO_MODEL; test in app
  • Day 4: Optional — 2nd training pass with eval feedback (active learning)
  • Day 5: Write model card (config, eval results, limitations)
